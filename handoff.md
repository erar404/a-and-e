# Handoff

## Goal
Build and refine the "surprise monthsary letter" feature for this static romantic gift site ("Walong Buwan" / "Usap Tayo") and, separately, make live video/audio calls in `chat.html` minimizable so the chat stays usable underneath.

Sub-goals across this session, in order:
1. **Ship the monthsary letter system to every page** (it previously only half-existed on `chat.html`, unstyled, and not on `index.html` at all): appear automatically at PH midnight on the 11th of each month, feel like it was "personally handed" over, and stay reachable/reopenable on every page for the rest of that day (today's target: July 11, 2026, the 9th monthsary — "Siyam na Buwan").
2. **Redesign the closed-letter interaction to fly around the screen**: instead of a docked tab, the closed envelope should visibly drift/roam the live page and be tappable anywhere to open, per explicit user request.
3. **Make live calls minimizable**: user confirmed (via clarifying question) they want an in-page draggable bubble that shrinks the call overlay's real box so the chat underneath (messages, composer) becomes usable again mid-call — NOT actual cross-page navigation persistence (ruled out because Document Picture-in-Picture only works on desktop Chrome/Edge, and the primary user is on her phone per `PRODUCT.md`).

Constraints established: vanilla HTML/CSS/JS, no build step, no frameworks (Docker+nginx static site). Phone-first performance: animations must be transform/opacity-driven. `prefers-reduced-motion` must have a real functional fallback, not just "turn off animation." Register is "brand" (romantic, dim/cinematic keepsake-box aesthetic — wine/rose/gold palette, Cormorant Garamond + La Belle Aurore fonts, explicitly NOT corny/clip-art). No em dashes were requested in this session but general project style avoids them in code comments where noted.

### 1. Monthsary letter system (flying envelope)
- A sealed envelope (`#monthsary-envelope`, small `.envelope-box` with `.envelope-flap` + `.envelope-seal`) now exists on **all three pages**: `index.html`, `chat.html`, `call-check.html`.
- It's hidden by default. `monthsary-timer.js` shows it (`startFlying()`) only when PH-local date is the 11th of the month (or via `?monthsaryTestSeconds=N` debug override), and hides it again once the day rolls over.
- While shown, it continuously drifts around the viewport via a 27s CSS keyframe loop (`envelope-fly`, 9 waypoints) plus a small flutter/bob on the inner box — pure `transform`/`opacity`, no layout properties animated.
- A one-time caption (`#monthsary-fly-hint`) fades in/out over ~7s the first time it appears, cueing that it's tappable.
- Tapping it anywhere mid-flight: JS freezes its current `getComputedStyle().transform` as an inline style, removes the CSS animation, forces a reflow, adds `.landing`, then on the next frame sets `transform: translate(-50%, -50%)` to fly it to dead center via a CSS transition (`0.85s cubic-bezier`). A backdrop (`#monthsary-backdrop`) fades in concurrently. After ~900ms (`LAND_MS` in JS, must stay ≥ the CSS transition duration), `.opened` is added, triggering the wax-seal-crack + flap-open keyframes, and `#monthsary-letter-card` (a separate fixed, center-anchored element, not nested inside the envelope) unhides and animates in (`letter-emerge`).
- Closing (✕ button or backdrop click) hides the card/backdrop and resets the envelope back to flying state (not gone for the day — it should keep drifting so it's reachable again).
- Letter content comes from `static/data/monthsary.json` via `months["N"].letter`, falling back to `defaults.letter` if absent. **The user has since written real content for month 9** (`months["9"]`, lines ~67–81 of that file) — this was a gap I flagged at the end of an earlier turn ("it'll show a generic placeholder unless you add month 9"), and the user filled it in themselves between turns. No action needed there now.
- `prefers-reduced-motion` fallback: envelope docks statically bottom-right (no flying), opening becomes a plain opacity crossfade instead of the fly/crack/emerge sequence.
- **Production bug fixed along the way**: `Dockerfile`'s hardcoded `COPY` file list was missing `monthsary-timer.js` entirely — the whole feature would have 404'd in the deployed container. Fixed (see `Files Actively Being Edited`).
- Two earlier design iterations were fully built then **entirely replaced** within this session (not just tweaked) — see `Failed Attempts`. The current code has zero references to either earlier approach; this was verified with `grep` for leftover class/id names (`monthsary-tab`, `envelope-hand`, `monthsary-letter-overlay`, `letterOverlay`) and confirmed clean.

### 2. Minimizable video/audio calls (`chat.html` only)
- New "⌄" minimize button in `.call-controls`, shown only once `callState === "active"` (not during ringing/dialing).
- Clicking it adds `.minimized` to `#call-overlay`, which **actually shrinks the overlay's real box** (not a visual-only trick) from `inset:0` full-screen to a small `clamp(108px, 30vw, 150px)` wide, `aspect-ratio: 3/4` rounded bubble, positioned via CSS custom properties `--call-min-x`/`--call-min-y` (JS-driven), defaulting to top-right just under the header (chosen specifically to clear the composer bar, which grows with textarea content — bottom-right was tried first and rejected for this reason, see below).
- The bubble is draggable via Pointer Events (`pointerdown`/`pointermove`/`pointerup` on `#call-overlay` itself, with `setPointerCapture`), clamped to viewport bounds, re-clamped on `resize`. A move under 4px is treated as a tap (not a drag) and triggers `restoreCall()` — full expand-back.
- While minimized: `.call-info`, `.call-controls` (full row), `.call-video-pip` (self-view), and `#call-monthsary-timer` are all `display:none` via `.call-overlay.minimized` descendant selectors. A separate compact control set (`.call-mini-controls`: mute + hangup, `.call-mini-expand`: explicit ⤢ icon) is shown/hidden via **JS toggling the `hidden` attribute directly** (not CSS class-based display) — this matters because this codebase has a global `[hidden] { display: none !important; }` rule, so any competing non-`!important` CSS display rule would silently lose; this was caught and fixed during the session.
- Mute state is synced between the full-screen mute button and the mini mute button via a shared `toggleMute()` function (previously inline duplicated logic, refactored out).
- Every new call (`showCallUI()`) and every call end (`hideCallUI()`) calls `restoreCall()` to guarantee minimize state never leaks between calls.
- No changes made to `call-check.html` or `index.html` for this feature — calls only happen in `chat.html`.

## Failed Attempts
- **First monthsary letter design (session start)**: a centered modal overlay (`.monthsary-letter-overlay`, `display:grid; place-items:center`) containing a sealed envelope that arrived via a "hand-off" entrance animation — a hand-drawn gold SVG line-art hand gesture (`.envelope-hand`, hand-crafted bezier paths) appeared to offer the envelope upward, held, then withdrew, followed by tap-to-crack-open. Access was via a docked ribbon tab (`.monthsary-tab`) pinned to the right edge of the screen, visible only on the monthsary day. **Why replaced**: not a failure exactly, but the user's next request ("make the closed letter look like it is flying around on the screen") explicitly asked for a different interaction model (continuous roaming vs. a static docked tab + centered modal). This entire design was removed and replaced, not iterated on — the hand SVG, the tab, and the `grid-area:1/1`-based overlay centering approach are all gone from the current code.
- **Grid-based envelope/card centering (`grid-area: 1/1` on both elements inside a `display:grid` overlay)**: worked for the modal design but was abandoned when the envelope became a free-roaming `position:fixed` element — grid layout doesn't apply to fixed-position children anyway, so this was restructured to independent `position:fixed; left:50%; top:50%;` + `transform` centering for both the envelope and the letter card, so they visually land at the exact same point without needing a shared grid parent.
- **Default minimized-call-bubble position at bottom-right**: initially placed the bubble at `bottom: 1rem` (mirroring where a "close corner" would intuitively go). **Why changed before shipping**: reasoned through that the composer bar (which can grow up to 140px tall with a multi-line textarea) is also anchored to the bottom of the viewport, so a bottom-right bubble would very likely sit on top of the send button/textarea. Changed to top-right, computed at runtime from the actual header element's `getBoundingClientRect().bottom` — this was caught during self-review, not from a user report, so it has **not been visually confirmed** to look right either; worth double-checking on a real phone screen.
- **`.call-mini-controls`/`.call-mini-expand` visibility via CSS class-based `display` rules** (e.g., `.call-overlay.minimized .call-mini-controls { display:flex }`): would have silently never worked, because these elements also carry the `hidden` HTML attribute by default, and this codebase has a global `[hidden] { display:none !important; }` rule that always wins over a non-`!important` `display:flex`. Caught during planning (not by running code, since no browser was available) and fixed by switching to JS-driven `.hidden = true/false` toggling instead, matching the pattern already used everywhere else in this file (`banner.hidden`, `callTimer.hidden`, etc.).
- **`envelope-hand` SVG z-index/stacking reasoning**: spent significant effort reasoning through CSS stacking-context rules (whether `position:relative` without explicit `z-index` creates a stacking context, how negative `z-index` on a descendant escapes to the nearest ancestor stacking context, etc.) to get the hand graphic to render behind the envelope but above the modal backdrop. This entire subsystem was later deleted when the design changed (see first Failed Attempt above), so this reasoning is now moot — flagging only so a future session doesn't waste time hunting for `.envelope-hand` expecting it to still exist.

## Context & Gotchas
- **No browser/screenshot tool was available in this environment at any point in this session.** All verification was: `node --check` for JS syntax, a hand-rolled Node script for HTML tag-balance, `grep -o '{' | wc -l` vs `}` count for CSS brace-balance, and a local `http.createServer` Node script to confirm files serve with 200s. This is a meaningfully weaker verification bar than actually rendering the page — treat all "it should work" claims in this handoff as reasoned-through, not confirmed.
- **This codebase intentionally duplicates shared CSS/HTML fragments across pages** (`styles.css` vs `chat.css` both have near-identical `:root` variables and monthsary CSS; `index.html`/`chat.html`/`call-check.html` all have near-identical monthsary markup blocks). This is the existing project convention (no build step, no includes/partials mechanism), not something introduced this session — when editing monthsary or shared-atmosphere styling in the future, remember to change it in **both** CSS files and check whether it needs to change in **all three** HTML files.
- **The `[hidden] { display: none !important; }` global rule** (in both `styles.css` and `chat.css`) always wins over any competing CSS `display` rule regardless of specificity. Any new show/hide UI state in this codebase should be driven via the `hidden` DOM attribute in JS, not via toggling a CSS class that sets `display`. This bit the call-mini-controls implementation once already (see Failed Attempts) and will bite anything else that forgets it.
- **PH timezone handling**: `monthsary-timer.js` computes "today" in `Asia/Manila` via `Intl.DateTimeFormat` with `timeZone: "Asia/Manila"`, not the browser's local timezone — this matters if testing from a non-PH machine, since the visible "today" in devtools/console may differ from what the script considers "today." The monthsary start date is `2025-10-11`; July 11, 2026 computes to month count 9 ("Siyam na Buwan").
- **`?monthsaryTestSeconds=N`** is the existing debug mechanism (predates this session) for forcing the countdown to fire N seconds from page load, without waiting for real midnight. Still fully supported by the rewritten JS.
- **The user is actively editing files between turns** (confirmed via `git diff` showing `static/data/monthsary.json` changed with content I never wrote — the user added the real month-9 letter themselves). Don't assume the working tree only reflects Claude's edits; check `git diff`/`git status` at the start of a resumed session rather than trusting this handoff's file list alone.
